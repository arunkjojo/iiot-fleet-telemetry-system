# Running the IIoT Fleet Telemetry System with Docker

## What this stack is

`docker-compose up --build` starts a **4-service stack** — `db`, `backend`, `frontend`,
`iiot-emitter` — and by default it runs on **live telemetry generated by a real Python IIoT
emitter**, not a dummy in-memory simulation.

> **This is the single most important behavior change since Sprint 01.** The committed
> `docker-compose.yml` sets `USE_LIVE_TELEMETRY=true` on the `backend` service, and the
> `iiot-emitter` service simulates the full 10,000-vehicle fleet, each vehicle posting its own
> telemetry reading on an independent tick loop to `POST /api/telemetry/ingest`. That data is
> persisted to PostgreSQL and rebroadcast to the dashboard over the existing SignalR hub
> (`/fleethub`). The frontend requires zero code changes to display it.
>
> The legacy in-memory `TelemetrySimulationService` (random-walk dummy data, no DB, no HTTP)
> still exists in the codebase and is only used when `USE_LIVE_TELEMETRY=false` — e.g. plain
> `dotnet run` outside Docker, where it remains the default. See
> [Switching back to dummy/simulated data](#switching-back-to-dummysimulated-data) below.

---

## Architecture overview

```
                     POST /api/telemetry/ingest
  iiot-emitter  ───────────────────────────────►  backend  ◄─────────────  frontend
  (Python,                                        (.NET 8,                 (Next.js 15,
   10,000 async                                    SignalR /fleethub,       browser dashboard)
   vehicle tasks)                                  buffered EF Core
                                                    writer)
                                                       │
                                                       ▼
                                                      db
                                                 (PostgreSQL 16,
                                                  tuned image)
```

- `iiot-emitter` → `backend`: outbound-only HTTP client. One `POST` per vehicle per tick.
  No inbound ports, no healthcheck of its own — ordering is via `backend`'s
  `service_healthy` condition.
- `backend` ← `db`: EF Core / Npgsql connection pool. Reads/writes `vehicles`,
  `telemetry_snapshots`, `vehicle_logs`.
- `frontend` → `backend`: REST calls (`/api/vehicles*`) plus a persistent SignalR connection
  to `/fleethub` for live push updates.
- `backend` never calls `frontend` or `iiot-emitter`.

## Quick start

Bring up the full stack, building images first:

```bash
docker-compose up --build
```

Detached (background) mode:

```bash
docker-compose up --build -d
```

Once the stack is healthy, these URLs become available:

| Service | URL | Notes |
|---|---|---|
| Frontend dashboard | http://localhost:3000 | Live-updating vehicle dashboard |
| Backend API | http://localhost:8080 | REST endpoints under `/api/**` |
| Swagger UI | http://localhost:8080/swagger | Interactive API docs |
| SignalR hub | http://localhost:8080/fleethub | WebSocket endpoint, MessagePack protocol |
| PostgreSQL | localhost:5432 | `fleet_telemetry` database, user `postgres` |

Give the stack roughly 30–60 seconds after `up` before expecting `telemetry_snapshots` to be
growing — `db` must pass its healthcheck before `backend` starts, and `backend` must pass its
healthcheck before `iiot-emitter` starts posting.

## Per-service breakdown

### `db` — PostgreSQL 16 (tuned)

- **Dockerfile:** `db/Dockerfile` — `FROM postgres:16-alpine`, copies in a custom
  `db/postgresql.conf` and launches Postgres with `-c config_file=/etc/postgresql/postgresql.conf`.
  The official Postgres entrypoint is otherwise untouched, so `POSTGRES_DB` /
  `POSTGRES_USER` / `POSTGRES_PASSWORD`-driven first-run initialization still works exactly as
  with the stock image.
- **Why a custom image:** `TelemetryPersistenceService` on the backend performs sustained,
  write-heavy batched inserts (up to ~2,000 rows/batch, roughly once a second). `postgresql.conf`
  tunes `max_connections` (200, up from the default 100), `shared_buffers` (256MB),
  `work_mem` (8MB), `maintenance_work_mem` (128MB), `effective_cache_size` (768MB), and
  `checkpoint_completion_target` (0.9, smoothing checkpoint I/O) for this workload.
  `listen_addresses = '*'` is also set explicitly — the stock default of `localhost` would make
  this instance unreachable from sibling containers on the Docker network (see
  [Troubleshooting](#c-db-unreachable-from-sibling-containers) below).
- **Schema ownership:** unchanged. This image never touches schema — EF Core migrations remain
  the only schema authority.
- **Env vars:** see the [full table](#full-environment-variable-reference) below.
- **Volume:** `postgres_data:/var/lib/postgresql/data` — persists across `docker-compose down`
  (but not `docker-compose down -v`).
- **Resource note:** default tuning assumes at least ~1GB of memory available to the container.

### `backend` — ASP.NET Core 8 Web API

- **Dockerfile:** `backend/Dockerfile` — multi-stage build, `mcr.microsoft.com/dotnet/sdk:8.0`
  for restore/publish, `mcr.microsoft.com/dotnet/aspnet:8.0` for the runtime image. Listens on
  port 8080 inside the container (`ASPNETCORE_URLS=http://+:8080`).
- **Responsibilities:** Swagger UI, the `/api/vehicles*` and `/api/telemetry/ingest` REST
  endpoints, the `/fleethub` SignalR hub, and (when `USE_LIVE_TELEMETRY=true`) the
  `TelemetryPersistenceService` buffered writer and `LiveBroadcastService` relay.
- **Env vars:** see the [full table](#full-environment-variable-reference) below.
  `USE_LIVE_TELEMETRY=true` is set in the committed `docker-compose.yml` — this is what makes
  the containerized stack live-by-default even though local `dotnet run` (no env var set)
  still defaults to the dummy simulation.
- **Healthcheck:** the `aspnet:8.0` runtime image ships neither `curl` nor `wget`, so the
  healthcheck uses bash's built-in `/dev/tcp` to confirm Kestrel is accepting connections on
  8080 (`bash -c 'exec 3<>/dev/tcp/localhost/8080'`).

### `frontend` — Next.js 15 (App Router)

- **Dockerfile:** `frontend/Dockerfile` — multi-stage build on `node:18-alpine`
  (`deps` → `builder` → `runner`). Builds a production Next.js app and serves it with
  `npm run start` on port 3000.
- **Requires `frontend/public/`** to exist at build time (`COPY --from=builder /app/public
  ./public`) — see [Troubleshooting](#d-frontend-image-fails-to-build-missing-public) if this
  directory is ever deleted.
- **Env vars:** `NEXT_PUBLIC_API_URL=http://backend:8080` — resolves the backend by its
  Compose service name over the shared Docker network, not `localhost`.
- **Healthcheck:** `node:18-alpine` ships busybox `wget` but not `curl`, so the healthcheck uses
  `wget -q --spider http://localhost:3000/`.

### `iiot-emitter` — Python IIoT fleet simulator

- **Dockerfile:** `iiot-emitter/Dockerfile` — `FROM python:3.12-slim`, installs
  `iiot-emitter/requirements.txt` (pinned `aiohttp==3.9.5`), runs `python emitter.py`.
  `PYTHONUNBUFFERED=1` is set so `docker-compose logs -f iiot-emitter` shows log lines
  immediately instead of being stdout-buffered.
- **What it does (`emitter.py`):**
  1. Fetches the canonical vehicle roster from `GET {BACKEND_URL}/api/vehicles/metadata`
     (retrying with a fixed backoff for up to a minute, in case the backend isn't accepting
     traffic yet outside of Compose's `service_healthy` gating).
  2. Slices the roster to `min(VEHICLE_COUNT, len(roster))` — it never assumes 10,000 vehicles
     exist; it simulates however many the roster actually has, capped at `VEHICLE_COUNT`.
  3. Spawns one independent `asyncio` task per vehicle. Each task staggers its first POST with
     a random `0..TICK_INTERVAL_SECONDS` delay, then loops: evolve local state incrementally
     (fuel mostly drains, speed/temp/engine health do small bounded random walks, position
     takes small bounded steps within the SF bounding box, cargo rarely changes), build the
     JSON payload, `POST /api/telemetry/ingest`, sleep `TICK_INTERVAL_SECONDS`, repeat.
  4. All outbound requests share one `aiohttp.ClientSession` backed by a
     `TCPConnector(limit=MAX_CONCURRENCY)`, plus an explicit `asyncio.Semaphore(MAX_CONCURRENCY)`
     around each POST — this is what bounds the emitter to at most `MAX_CONCURRENCY` in-flight
     HTTP requests regardless of `VEHICLE_COUNT`.
  5. A per-vehicle `try/except` around every POST catches `aiohttp.ClientError` and
     `asyncio.TimeoutError`; failures are logged and counted, never raised — one vehicle's
     failed tick never crashes the process.
  6. A summary log line (`ticks_sent`, `errors`) is emitted roughly every 10 seconds.
  7. `SIGINT`/`SIGTERM` set a shared `asyncio.Event`; all vehicle tasks and the summary task
     observe it and exit within one tick, so `docker-compose stop iiot-emitter` and `Ctrl+C`
     both shut down cleanly.
- **No inbound ports, no healthcheck** — it's an outbound-only client; Compose orders it via
  `backend: condition: service_healthy`.
- **Env vars:** see the [full table](#full-environment-variable-reference) below.
- **Resource note:** the committed default (`VEHICLE_COUNT=10000`) runs 10,000 concurrent
  `asyncio` tasks. This is intentionally the full fleet — see
  [Scaling the emitter down](#scaling-the-emitter-down-for-a-lower-spec-machine) if your machine
  can't comfortably sustain that alongside `db`/`backend`.

## Full environment variable reference

| Variable | Service | Default (as committed) | Description |
|---|---|---|---|
| `POSTGRES_DB` | `db` | `fleet_telemetry` | Database created on first run |
| `POSTGRES_USER` | `db` | `postgres` | Superuser created on first run |
| `POSTGRES_PASSWORD` | `db` | `Admin123.` | Password for `POSTGRES_USER` |
| `ASPNETCORE_ENVIRONMENT` | `backend` | `Production` | ASP.NET Core hosting environment |
| `FRONTEND_ORIGIN` | `backend` | `http://frontend:3000` | Primary CORS-allowed origin |
| `ADDITIONAL_FRONTEND_ORIGINS` | `backend` | `http://localhost:3000` | Comma-separated extra CORS origins |
| `ConnectionStrings__Fleet` | `backend` | `Host=db;Port=5432;Database=fleet_telemetry;Username=postgres;Password=Admin123.` | PostgreSQL connection string (Npgsql format) |
| `USE_LIVE_TELEMETRY` | `backend` | `true` (Docker Compose) / `false` (bare `dotnet run`) | Toggles live ingestion vs. dummy in-memory simulation as the vehicle data source |
| `NEXT_PUBLIC_API_URL` | `frontend` | `http://backend:8080` | Base URL the browser-served app uses to reach the backend |
| `BACKEND_URL` | `iiot-emitter` | `http://backend:8080` | Base URL the emitter posts telemetry to and fetches the roster from |
| `VEHICLE_COUNT` | `iiot-emitter` | `10000` | Number of vehicles simulated, capped to the roster size actually returned by `/api/vehicles/metadata` |
| `TICK_INTERVAL_SECONDS` | `iiot-emitter` | `3` | Seconds between telemetry ticks, per vehicle |
| `MAX_CONCURRENCY` | `iiot-emitter` | `300` | Max concurrent outbound POSTs (shared `TCPConnector` limit + semaphore) |
| `REQUEST_TIMEOUT_SECONDS` | `iiot-emitter` | `5` (script default; not set in `docker-compose.yml`) | Per-request aiohttp timeout |

All five `iiot-emitter` variables, and `USE_LIVE_TELEMETRY`, are also documented in
`docs/requirements/REQUIREMENTS.md` section 9 (Environment Variables) as the source of truth.

## Switching back to dummy/simulated data

To run the containerized stack against the legacy in-memory `TelemetrySimulationService`
instead of live ingestion, override `USE_LIVE_TELEMETRY` to `false` on the `backend` service.
Two ways to do this without editing the committed `docker-compose.yml`:

**Option A — inline env override on `up`:**

```bash
USE_LIVE_TELEMETRY=false docker-compose up --build -d backend frontend db
```

(Note: setting a shell env var only overrides a `docker-compose.yml` value that references
`${USE_LIVE_TELEMETRY}` — since the committed file hardcodes `USE_LIVE_TELEMETRY=true` under
`backend.environment`, prefer Option B below for a reliable override.)

**Option B — `docker-compose.override.yml` (recommended):**

```yaml
# docker-compose.override.yml (not committed — local only)
services:
  backend:
    environment:
      - USE_LIVE_TELEMETRY=false
```

`docker-compose` automatically merges `docker-compose.override.yml` on top of
`docker-compose.yml`. With this in place you'll usually also want to stop `iiot-emitter`, since
it has nothing to post against in dummy mode:

```bash
docker-compose up --build -d db backend frontend
```

In dummy mode, `backend` seeds and simulates 10,000 vehicles in-memory (no DB writes, no
ingestion endpoint traffic) exactly as it did before this sprint.

## Scaling the emitter down for a lower-spec machine

The committed default is the **full 10,000-vehicle fleet, intentionally** — that's the sprint's
target scale. If your machine can't comfortably run 10,000 concurrent `asyncio` tasks alongside
`db` and `backend`, override `VEHICLE_COUNT` (and optionally `TICK_INTERVAL_SECONDS` /
`MAX_CONCURRENCY`) rather than editing `docker-compose.yml` directly.

**Option A — `docker-compose.override.yml`:**

```yaml
# docker-compose.override.yml (not committed — local only)
services:
  iiot-emitter:
    environment:
      - BACKEND_URL=http://backend:8080
      - VEHICLE_COUNT=100
      - TICK_INTERVAL_SECONDS=3
      - MAX_CONCURRENCY=50
```

```bash
docker-compose up --build -d
```

**Option B — recreate just the emitter with an ad-hoc override:**

```bash
docker-compose up --build -d db backend frontend
docker-compose run --rm -e VEHICLE_COUNT=100 -e BACKEND_URL=http://backend:8080 iiot-emitter
```

(`docker-compose run` starts a one-off, non-daemonized container attached to your terminal —
useful for quick local testing; use `up -d` with an override file for a persistent lower-spec
run.)

The emitter always fetches the real roster first and slices it to
`min(VEHICLE_COUNT, len(roster))`, so any value from `1` up to `10000` works without code
changes.

## Common operational commands

Build only, without starting anything:

```bash
docker-compose build
docker-compose build backend        # single service
```

View logs:

```bash
docker-compose logs -f              # all services, follow
docker-compose logs -f iiot-emitter # just the emitter
docker-compose logs -f backend
docker-compose logs --tail=50 db
```

Restart a single service:

```bash
docker-compose restart backend
```

Stop the stack (containers stop, volumes and images persist):

```bash
docker-compose down
```

Stop and wipe volumes (drops the `postgres_data` volume — full data reset):

```bash
docker-compose down -v
```

Exec into a running container:

```bash
docker-compose exec backend bash
docker-compose exec db sh          # postgres:16-alpine is Alpine-based; use sh, not bash
```

Run `psql` against the database from the host (via the `db` container):

```bash
docker-compose exec db psql -U postgres -d fleet_telemetry
```

Check table row counts (useful for confirming live ingestion is actually writing data):

```bash
docker-compose exec -T db psql -U postgres -d fleet_telemetry -c \
  "SELECT COUNT(*) FROM vehicles;"

docker-compose exec -T db psql -U postgres -d fleet_telemetry -c \
  "SELECT COUNT(*) FROM telemetry_snapshots;"

docker-compose exec -T db psql -U postgres -d fleet_telemetry -c \
  "SELECT COUNT(*) FROM vehicle_logs;"
```

Watch `telemetry_snapshots` grow in real time (run twice, a few seconds apart, and compare):

```bash
docker-compose exec -T db psql -U postgres -d fleet_telemetry -tAc \
  "SELECT COUNT(*) FROM telemetry_snapshots;"
```

## Manual API test — POST a sample telemetry reading

`POST /api/telemetry/ingest` accepts a single vehicle's reading (`TelemetryIngestRequest`),
computes its status server-side, upserts the live store, enqueues a durable DB write, and
returns `202 Accepted`. Field names are camelCase and must match exactly:

```bash
curl -s -w "\n%{http_code}\n" -X POST http://localhost:8080/api/telemetry/ingest \
  -H "Content-Type: application/json" \
  -d '{
    "vehicleId": "VEH-00001",
    "driverName": "Joy",
    "model": "NV Cargo",
    "latitude": 37.77,
    "longitude": -122.41,
    "fuelPercent": 84.5,
    "speedKph": 45.2,
    "engineHealth": 72,
    "tempCelsius": 62,
    "cargoLoad": 3200
  }'
```

Expected response: HTTP `202` with a body like:

```json
{ "status": "accepted", "vehicleId": "VEH-00001", "computedStatus": "active" }
```

`recordedAtUtc` is optional in the payload — omit it and the server stamps `DateTime.UtcNow`.

**Important:** `vehicleId` must be one that already exists in the `vehicles` table. Get the
full, valid list from:

```bash
curl -s http://localhost:8080/api/vehicles/metadata | python -m json.tool
```

Posting an unknown `vehicleId` is accepted at the HTTP layer (`202`) but the resulting
`telemetry_snapshots` row insert will fail its foreign-key constraint when
`TelemetryPersistenceService` flushes the batch — the error is logged and that batch item is
silently dropped, not persisted. See
[Troubleshooting: FK violations](#b-fk-violations-from-posting-an-unknown-vehicleid) below.

## Troubleshooting

### (a) Emitter can't reach the backend

Symptoms: `iiot-emitter` logs repeated `"Roster fetch attempt N/30 failed"` warnings, or the
container exits with `RuntimeError: Could not fetch vehicle roster ... after 30 attempts`.

1. Check the emitter actually waited for `backend` to be healthy:
   `docker-compose ps backend` should show `Up (healthy)`. `iiot-emitter` has
   `depends_on: backend: condition: service_healthy` — if `backend` never becomes healthy,
   Compose won't start `iiot-emitter` at all, or will keep it waiting/restarting.
2. Confirm `BACKEND_URL` resolves inside the Docker network. It must be the **Compose service
   name**, not `localhost`: `http://backend:8080`, not `http://localhost:8080`. `localhost`
   inside the `iiot-emitter` container refers to that container itself.
3. Confirm the backend is actually listening: `docker-compose logs backend | tail -50` and look
   for Kestrel startup lines / unhandled startup exceptions (e.g. a bad
   `ConnectionStrings__Fleet`).
4. From inside the emitter container, test connectivity directly:
   `docker-compose exec iiot-emitter python -c "import urllib.request;
   print(urllib.request.urlopen('http://backend:8080/api/vehicles/metadata').status)"`

### (b) FK violations from posting an unknown `vehicleId`

Symptoms: `telemetry_snapshots` row count isn't growing even though `iiot-emitter` (or a manual
`curl`) reports successful `202` responses; `backend` logs show a foreign-key constraint error
around `SaveChangesAsync` in `TelemetryPersistenceService`.

- The ingest endpoint returns `202` as soon as the request is validated and queued — it does
  **not** wait for the DB write, so a `202` does not guarantee persistence.
- `vehicle_id` in both `telemetry_snapshots` and `vehicle_logs` is a foreign key into `vehicles`.
  Any `vehicleId` not already present in that table will fail the batched insert; the failure is
  caught, logged, and the offending batch item dropped — the persistence loop itself does not
  crash.
- Fix: only ever post `vehicleId`s obtained from `GET /api/vehicles/metadata` (this is exactly
  what `emitter.py` does — it never invents IDs). If you're testing manually with `curl`, fetch
  the roster first.

### (c) `db` unreachable from sibling containers

Symptoms: `backend` fails to start / crashes on startup with a connection-refused or
connection-timeout error against `db`, even though `docker-compose ps db` shows healthy.

- This was previously an issue because Postgres's stock default `listen_addresses = 'localhost'`
  only accepts connections from within the `db` container itself — unreachable from `backend`
  or `frontend` on the shared `iiot-fleet-net` network.
- Fixed in `db/postgresql.conf`: `listen_addresses = '*'` is set explicitly. If you customize
  `db/postgresql.conf` further, do not drop this line, or sibling-container connectivity breaks
  again.
- If you still see this, confirm `db/Dockerfile` is actually being used
  (`docker-compose.yml`'s `db` service should read `build: ./db`, not `image: postgres:16-alpine`)
  and rebuild: `docker-compose build --no-cache db`.

### (d) Frontend image fails to build (missing `public/`)

Symptoms: `docker-compose up --build` (or `docker-compose build frontend`) fails during the
`runner` stage with an error copying `/app/public` — `frontend/Dockerfile` has
`COPY --from=builder /app/public ./public`, which requires `frontend/public/` to exist in the
build context.

- This directory was missing at one point during Sprint 02 and blocked the frontend build
  entirely; it has been restored and is now tracked in the repo.
- If this happens again: confirm `frontend/public/` exists and is not excluded by
  `frontend/.dockerignore` or `.gitignore`, then rebuild:
  `docker-compose build --no-cache frontend`.
- All other services (`db`, `backend`, `iiot-emitter`) do not depend on `frontend` and can be
  brought up independently while this is being fixed:
  `docker-compose up --build -d db backend iiot-emitter`.

## Custom ports

To change host port mappings, edit the `ports:` entries in `docker-compose.yml` (e.g.
`"3000:3000"` → `"3001:3000"` for the frontend). Container-internal ports (8080 for the
backend, 3000 for the frontend, 5432 for Postgres) should not be changed without also updating
the corresponding `ENV`/config values baked into each image.
