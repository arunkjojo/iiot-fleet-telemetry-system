## Context

Sprint 01 added PostgreSQL, EF Core, and Swagger scaffolding, but the live data path was never actually wired up: `VehiclesController`, `LogsController`, and `MetadataController` still read exclusively from `TelemetrySimulationService`'s in-memory `ConcurrentDictionary`, which is synthetic random-walk data. The `vehicles`, `telemetry_snapshots`, and `vehicle_logs` tables exist and are seeded but are never written to by live traffic. This sprint replaces the dummy source with a real ingestion pipeline: a standalone Python script emits telemetry for 10,000 simulated IIoT-equipped vehicles over HTTP to a new backend ingestion endpoint, which persists it to PostgreSQL and rebroadcasts it over the existing SignalR hub so the frontend requires zero changes to show it live. The legacy in-memory simulation is preserved (per `backend/AGENTS.md`, `TelemetrySimulationService.cs` must never gain DB or HTTP calls) and is switched off via an environment toggle rather than deleted, so local `dotnet run` development without Docker still works out of the box.

**Decisions locked in before this sprint was authored (do not re-litigate without user sign-off):**
1. `TelemetrySimulationService` stays in the codebase untouched; a `USE_LIVE_TELEMETRY` env var (default `false`) decides at startup whether the sim or the live ingestion pipeline runs. Docker Compose sets it to `true`.
2. The emitter simulates the full fleet: 10,000 vehicles, each an independent `asyncio` task, bounded by a concurrency semaphore so the backend and DB are not hit by 10,000 simultaneous connections.
3. Transport is one HTTP `POST` per vehicle per tick to a new `POST /api/telemetry/ingest` endpoint — not a batch endpoint, not a direct SignalR client.
4. DB write load is decoupled from emitter cardinality: the ingest endpoint enqueues into an in-process buffer; a single background service drains it in batches. The number of emitters never equals the number of DB connections.

**Implementation notes:**

1. `TelemetryPersistenceService.ExecuteAsync` loop: every 1000ms (or when a batch reaches 2,000 items, whichever first), drain both channels into `List<T>`, call `db.TelemetrySnapshots.AddRangeAsync(...)` / `db.VehicleLogs.AddRangeAsync(...)`, then one `SaveChangesAsync()` per entity type. Use `IServiceScopeFactory` to create a scoped `FleetDbContext` per flush (the service itself is a singleton hosted service; `DbContext` is scoped).
2. The controller must NOT call `SaveChangesAsync` synchronously — it only writes to the channel (`await _channel.Writer.WriteAsync(entity)`) and returns immediately. This is what decouples emitter count from DB connection count.
3. `VehicleId` FK safety: if `TelemetrySnapshots.AddRangeAsync` fails on FK violation (vehicle_id not in `vehicles` table — e.g., emitter misconfigured), log the error and drop that batch item; do not crash the persistence loop. Wrap `SaveChangesAsync` in try/catch.
4. Status-change detection for `VehicleLogEntity`: before calling `Upsert`, read the previous value via `ILiveTelemetryStore.TryGet(vehicleId, out var previous)`. If `previous == null` (first-ever reading) log `INFO "Vehicle online — live telemetry started"`. If `previous.Status != newStatus`, log `WARN "STATUS_CHANGE {previous} -> {new}"`. Also log `ERROR`/`WARN` for the same threshold crossings `TelemetrySimulationService` already logs (overspeed, low fuel, high temp) to keep `LogsController` output meaningful.
5. Channel capacity 50,000 with `BoundedChannelFullMode.DropOldest` — under sustained overload, prefer dropping the oldest buffered snapshot over blocking the HTTP request thread.
6. Response body: `{ "status": "accepted", "vehicleId": "...", "computedStatus": "warning" }` — useful for emitter-side debugging, not required by the frontend.
